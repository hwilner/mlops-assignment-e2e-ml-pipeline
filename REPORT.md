# MLOps Assignment: End-to-End ML Pipeline

This repository implements a complete, reproducible evaluation pipeline for coding agents using Apache Airflow and MLflow. It transforms ad-hoc shell scripts into a durable, observable, and parameterised workflow.

## Architecture

The pipeline is orchestrated by Airflow and consists of four sequential tasks defined in `dags/evaluate_agent.py`:

1. **`prepare_run`**: Validates Airflow parameters, generates a unique `run_id`, creates the `runs/<run-id>/` directory tree, and writes the canonical `config.json`.
2. **`run_agent`**: Executes `mini-swe-agent` via the parameterised `scripts/mini-swe-bench-batch.sh`. Outputs trajectories and `preds.json` to `runs/<run-id>/run-agent/`.
3. **`run_eval`**: Executes the SWE-bench evaluation harness via `scripts/swe-bench-eval.sh`. Outputs logs and per-instance `report.json` files to `runs/<run-id>/run-eval/`.
4. **`summarize_and_log`**: Parses the evaluation reports to calculate the resolve rate, writes a `metrics.json` and a `manifest.json` (which acts as a map to all artefacts), and logs the parameters, metrics, and artefact URI to MLflow.

### Production Deployment

The repository includes a `docker-compose.yaml` that deploys the full stack on a single VM:
- **PostgreSQL**: Shared metadata database for Airflow.
- **Airflow Webserver & Scheduler**: Orchestrates the DAGs.
- **MLflow**: Tracks experiment parameters, metrics, and artefact locations.

## How to Trigger the DAG

1. Start the stack: `docker compose up -d`
2. Open the Airflow UI at `http://localhost:8080` (admin/admin).
3. Unpause the `evaluate_agent` DAG.
4. Click the **▶ (Trigger DAG w/ config)** button.
5. Provide the configuration JSON. Example:
   ```json
   {
     "split": "test",
     "subset": "verified",
     "workers": 3,
     "model": "nebius/moonshotai/Kimi-K2.6",
     "task_slice": "0:3"
   }
   ```
6. Click **Trigger**.

## Artefact Layout

Every run produces a self-contained, reproducible directory structure:

```text
runs/<run-id>/
├── config.json          # The exact parameters used for this run
├── manifest.json        # Pointers to all artefacts and metrics summary
├── metrics.json         # Aggregated evaluation results (resolve rate, etc.)
├── run-agent/           # Agent outputs
│   ├── preds.json       # The final predictions submitted to SWE-bench
│   └── astropy__astropy-12907/  # Per-instance trajectory files
└── run-eval/            # Evaluation outputs
    └── test/nebius__moonshotai__Kimi-K2.6/astropy__astropy-12907/
        ├── eval.sh      # The evaluation script generated by SWE-bench
        ├── patch.diff   # The patch generated by the agent
        ├── report.json  # Pass/fail status for this instance
        └── run_instance.log # Full execution log
```

## MLflow Tracking

All runs are logged to MLflow (`http://localhost:5000`). The experiment name is derived from the model ID (e.g., `swebench-nebius__moonshotai__Kimi-K2.6`).

Each run logs:
- **Parameters**: `split`, `subset`, `workers`, `model`, `task_slice`, `cost_limit`.
- **Metrics**: `total_instances`, `resolved`, `unresolved`, `patch_applied`, `resolve_rate`.
- **Tags**: `run_id` and `artifact_uri` (pointing to the local `runs/<run-id>/` directory or an S3 bucket).

## Rerun Instructions

To reproduce a run:
1. Open the `manifest.json` or `config.json` from the run directory (or fetch the parameters from MLflow).
2. Trigger the `evaluate_agent` DAG in Airflow using the exact same JSON configuration.
3. The pipeline will execute deterministically (subject to the LLM's inherent randomness) and produce a new run directory with the results.
